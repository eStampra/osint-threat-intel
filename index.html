PROJECT 01 // PHASE 1 // WEEK 3–4
OSINT Investigation — Open Source Person Profile
Beginner ▼
Objective

Demonstrate ability to aggregate public information about an individual using only legal, public sources. Use a public figure (politician, CEO, or yourself) as the target. Document your methodology step-by-step.

Tools Used
Google Dorking Sherlock (username) theHarvester Wayback Machine LinkedIn OSINT Framework
Deliverable
  • → GitHub repo: osint-person-profile
  • → README.md with methodology
  • → findings.md (sanitised)
  • → Screenshots folder
  • → tools-used.md cheatsheet
Step-by-Step Instructions
  1. Choose target: Pick yourself, a public figure, or use a fictional persona from Trace Labs practice cases. Never investigate private individuals without consent.
  2. Passive recon — Google Dorking: Run these searches: "[name]" site:linkedin.com, "[name]" filetype:pdf, "[name]" -site:twitter.com. Record every query and result.
  3. Username enumeration: Install Sherlock on Kali. Run: python3 sherlock.py [username]. Screenshot output. Note which platforms have the account.
  4. Email discovery: Run theHarvester: theHarvester -d [domain] -b google. Also check Hunter.io free tier for domain emails.
  5. Historical data: Check Wayback Machine for old website versions. Use cached.google.com for deleted pages.
  6. Pivot chain: Map the connections: email → linked accounts → associated domains → related people. Draw this as a diagram (draw.io is free).
  7. Write findings report: Use this structure — Executive Summary (2 para), Data Sources Used, Key Findings (bullet points), Pivot Chain Diagram, Limitations, Recommendations. Sanitise any real personal data before committing.
  8. GitHub commit: Create repo, write README (what this is, tools used, methodology overview). Commit all files. Add a .gitignore to exclude any raw PII.
# GitHub repo structure for this project osint-person-profile/ ├── README.md # Overview, methodology, tools ├── findings.md # Sanitised findings report ├── methodology.md # Step-by-step process used ├── pivot-chain.png # Visual diagram of connections ├── screenshots/ # Evidence screenshots └── tools-cheatsheet.md # Commands used, for reference
PROJECT 02 // PHASE 2 // WEEK 6–7
Threat Actor Profile — APT Research Report
Intermediate ▼
Objective

Produce a professional Cyber Threat Intelligence (CTI) report on a known APT group. This is the core output of a threat intel analyst role. Use MITRE ATT&CK to map TTPs.

Tools Used
MITRE ATT&CK Navigator VirusTotal AbuseIPDB Shodan Any.Run OpenCTI (free)
Deliverable
  • → GitHub repo: apt-threat-profile
  • → Full CTI report (PDF + MD)
  • → ATT&CK Navigator layer file
  • → IOC list (CSV format)
Step-by-Step Instructions
  1. Choose APT group: Recommended starters: APT28 (Fancy Bear), Lazarus Group, TA505, or Kimsuky. All have extensive public reporting.
  2. Research sources: Read: (1) MITRE ATT&CK group page, (2) CISA advisory on this group, (3) at least one vendor report (Mandiant, CrowdStrike, or Recorded Future — all have free public reports). Take notes under: Origin, Targets, Motivation, TTPs, Known Tools.
  3. Map to ATT&CK Navigator: Go to mitre-attack.github.io/attack-navigator. Create a new layer. Highlight every technique used by your chosen group. Export as PNG and JSON.
  4. Collect IOCs: From public reports, extract: IP addresses, domains, file hashes, email addresses. Validate each one on VirusTotal and AbuseIPDB. Build a CSV: Type, Value, Confidence, Source.
  5. Write the CTI report using this structure: (1) Executive Summary, (2) Actor Overview, (3) Targeting & Victimology, (4) TTPs with ATT&CK IDs, (5) Indicators of Compromise, (6) Defensive Recommendations, (7) References.
  6. Format professionally: Export as PDF. Use clean markdown formatting. Add a header with TLP (Traffic Light Protocol) designation — use TLP:WHITE for public sharing.
  7. Commit to GitHub with a strong README explaining your methodology and sources.
PROJECT 03 // PHASE 2 // WEEK 8–9
Infrastructure OSINT — Domain & IP Investigation
Intermediate ▼
Objective

Investigate a suspicious domain or IP address using passive OSINT techniques. Determine ownership, hosting, historical DNS, related infrastructure, and threat indicators.

Tools Used
Shodan Censys VirusTotal SecurityTrails URLScan.io Whois PassiveDNS
Deliverable
  • → GitHub repo: infra-osint-investigation
  • → Investigation report (markdown)
  • → Timeline of domain/IP history
  • → Infrastructure diagram
  • → IOC table
Step-by-Step Instructions
  1. Choose a target domain/IP: Use a known malicious domain from a public threat feed (e.g., Abuse.ch, PhishTank) or take one from a public CISA advisory. Never scan or probe live systems actively.
  2. Passive DNS: Use SecurityTrails (free tier) or RiskIQ Community for historical DNS records. Record: all A records, MX records, when first/last seen, nameservers.
  3. Shodan search: Search the IP on Shodan. Note: open ports, banners, certificates, hosting provider, geolocation, related IPs on same ASN.
  4. Certificate transparency: Search crt.sh for SSL certificates issued to the domain. Find subdomains and related infrastructure.
  5. URLScan.io: Search the domain. Review screenshots, outgoing links, and technology fingerprinting.
  6. Pivot and cluster: From the IP, find all domains hosted there (reverse IP lookup). From those domains, check for shared certificates and hosting patterns.
  7. Write report and build infrastructure diagram in draw.io showing how all the pieces connect.
PROJECT 04 // PHASE 3 // WEEK 11–12
SIEM Log Analysis — Threat Hunting Simulation
Advanced ▼
Objective

Analyse simulated security logs to detect a simulated attack. Document the detection methodology, IOCs found, and write an incident report. Demonstrates core SOC analyst skill.

Tools Used
Splunk Free (local) LetsDefend Alerts Blue Team Labs Online Elastic SIEM (free)
Deliverable
  • → GitHub repo: siem-threat-hunt
  • → Detection report (markdown)
  • → SPL queries used (if Splunk)
  • → Attack timeline reconstruction
  • → Recommendations doc
Step-by-Step Instructions
  1. Set up Splunk Free: Download Splunk Enterprise free tier (500MB/day). Or use the free Splunk Attack Range lab datasets from GitHub (splunk/attack_range).
  2. Ingest sample logs: Use the Boss of the SOC (BOTS) dataset v1 from Splunk's GitHub. This is a real CTF dataset with simulated attacks baked in.
  3. Threat hunt scenario: Follow the BOTS v1 questions to guide your investigation. Find: web application attack, malware beaconing, lateral movement, data exfiltration.
  4. Write SPL queries and document each one with: what you were looking for, what you found, ATT&CK technique ID.
  5. Reconstruct the attack timeline: Build a chronological list of attacker actions with timestamps, source IPs, and techniques used.
  6. Write incident report: Executive summary, timeline, IOCs, MITRE ATT&CK mapping, containment recommendations.
PROJECT 05 // PHASE 3 // WEEK 13–14
YARA Rule Development — Malware Signature Writing
Advanced ▼
Objective

Write YARA rules to detect known malware families based on static signatures. YARA is a core skill for threat intelligence and malware analysis roles.

Tools Used
YARA Any.Run VirusTotal MalwareBazaar Cuckoo Sandbox
Deliverable
  • → GitHub repo: yara-rules-collection
  • → 5+ YARA rule files
  • → Test evidence screenshots
  • → Rule documentation (what each detects)
Step-by-Step Instructions
  1. Install YARA: On Kali Linux: sudo apt install yara. Read the YARA documentation (readthedocs) for rule syntax.
  2. Study existing rules: Browse the Awesome-YARA GitHub repo for examples. Read 10 existing rules and understand the structure: meta, strings, condition blocks.
  3. Get sample malware hashes: From MalwareBazaar (abuse.ch), find hashes for common families: Emotet, Cobalt Strike, AgentTesla. Download samples only in an isolated VM.
  4. Write a basic YARA rule: Start with string-based detection of a known malware family. Test with: yara rule.yar /path/to/samples.
  5. Iterate and improve: Write 5+ rules for different families. Add conditions, use hexadecimal strings for binary patterns, use regex for network IOCs.
  6. Document each rule: For every rule file, include: what it detects, how you derived the signature, test results, false positive risk.
PROJECT 06 // PHASE 4 // WEEK 17–18
Capstone — Full Threat Intelligence Report
Capstone ▼
This is your flagship portfolio piece. Dedicate 3 full days-off sessions to it. This is what you show in interviews.
Objective

Produce a publication-quality threat intelligence report on a current threat campaign. Combine all skills: OSINT, ATT&CK mapping, IOC collection, infrastructure analysis, YARA, and professional CTI writing.

Tools Used
All previous tools OpenCTI MISP Maltego ATT&CK Navigator
Deliverable
  • → Professional PDF report (15–20 pages)
  • → GitHub repo with all assets
  • → ATT&CK Navigator layer
  • → IOC CSV for SIEM import
  • → YARA detection rule
  • → Executive brief (1 page)
Report Structure (follow exactly)
  1. Cover page: Report title, TLP classification, date, author (your name), version number.
  2. Executive Summary (1 page): Who is the threat actor? What are they targeting? Why does it matter? What should the reader do?
  3. Threat Actor Overview: Origins, attribution confidence, aliases, historical activity timeline.
  4. Recent Campaign Analysis: Specific attack campaign from the last 12 months. Initial access, execution, persistence, exfiltration. Reference public reports.
  5. Technical Analysis — TTPs: Full ATT&CK mapping table. For each technique: ID, name, description of how this actor uses it, example from public reporting.
  6. Infrastructure Analysis: C2 servers, domains, hosting patterns. Include Shodan/Censys screenshots.
  7. Indicators of Compromise: Full IOC table with type, value, confidence, first seen, source.
  8. Detection & Hunting Guidance: Splunk/SIEM queries, YARA rule (include inline), network signatures.
  9. Recommendations: Specific, prioritised mitigations for each MITRE technique observed.
  10. References: All sources cited in ISAC format.
